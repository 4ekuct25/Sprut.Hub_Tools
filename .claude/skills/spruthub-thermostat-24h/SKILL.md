# Суточный отчёт термостата Sprut.Hub

## Где данные (главное)
Числовая история ВСЕХ показаний лежит в архиве **отладочной информации** хаба, НЕ в текстовых логах.

- В web.spruthub.ru карточка датчика показывает только текущее значение (графиков истории нет),
  а WebSocket-API отдаёт лишь текущее состояние — историю оттуда не достать.
- Архив: Настройки → меню «···» (вверху справа) → **Скачать отладочную информацию**
  → `~/Downloads/Sprut.Hub_DevInfo_*.zip` (~10 МБ).
- Внутри: `External/Bridges/History/1/history.duckdb` (+ `.wal` свежий буфер) и
  `segments/YYYY/MM/*.parquet` (по суткам). Таблица `History(timestamp_ms, a_id, s_id, c_id, type, value)`.
  Время в БД — UTC; локальное = +3 ч (MSK).
- Для разбора РЕШЕНИЙ сценария дополнительно: «Скачать логи» → `Sprut.Hub_Log_*.zip`,
  файл `External/spruthub.log`; строки сценария — тег `Automation`, фильтр `Logic_5_Service_42_13`
  (`Эмуляция: target=,current=,temp=N°C`, `реле охлаждения ON`, `Выключатель кондиционера: …`,
  `Проверка датчика: с последнего обновления N мин (timeout 60)`).

## Потоки истории (a_id · s_id · c_id)
- комната (внешний датчик SNZB-02D, вход термостата): `8 · 13 · 15`
- улица (LUMI weather): `17 · 13 · 15`
- датчик самого кондея (внутр., возвратный воздух): `22 · 17 · 21`
- целевая температура, которую сценарий шлёт в кондей: `22 · 17 · 22`
- скорость вентилятора кондея (1..3): `22 · 17 · 24`
- кабинетный VIOMI 9000 (если нужен): `23 · 17 · 21`
- **двери-герконы** (нагрузка на кондей): id=4 «Дверь кухня» (Aqara MCCGQ11LM, serial 00158D0009F5C5A7), также id=5/24. `c_id=15`, значение 1=открыто. Скрипт подсвечивает состояние двери агрегированно (бакеты по `--door-bucket` мин, порог `--door-thresh`; дефолт 30м/0.5 — частые открыл/закрыл схлопываются в основное состояние) и печатает % открытости за окно; набор — словарь `DOORS` вверху скрипта. ContactSensor шлёт ТОЛЬКО изменения — состояние реконструируется непрерывно (затравка последним событием + перенос через сутки), не фильтром по дню.

ID стабильны для этих устройств. Если хаб перепривязывали — проверить `python3 scripts/thermo24.py --list`
и поправить словарь `STREAMS` вверху скрипта.

## Грабли
- **Таймзона DuckDB:** фильтровать историю ПО СЫРЫМ мс (`where timestamp>{lo}`), не через `to_timestamp(ts/1000+off)` со сравнением с датой-строкой — не-UTC сессия сдвигает границы окна и режет события (был баг: фантом «дверь открыта до 00:00»). Скрипт ставит `SET TimeZone='UTC'` и конвертит в локаль только в Python.
- **Датчики-состояния (двери-герконы):** ContactSensor шлёт только изменения; состояние в момент X = последнее событие до X. Реконструировать непрерывно (затравка + перенос через сутки), не фильтром по дню.

## Как сделать отчёт
1. Убедиться, что в `~/Downloads` есть свежий `Sprut.Hub_DevInfo_*.zip`. Если нет/старый — скачать
   из хаба через браузер (Claude in Chrome): Настройки → «···» → «Скачать отладочную информацию»
   (и при необходимости «Скачать логи»). Внимание: плитка «Кондиционер» — это быстрый
   ВЫКЛЮЧАТЕЛЬ, не открывай её кликом.
2. Запустить скрипт (он сам берёт самый свежий DevInfo, выгружает 24 ч и строит единый график):
   ```
   pip install duckdb matplotlib --break-system-packages
   python3 scripts/thermo24.py                       # весь день, шкала 20–29°C
   python3 scripts/thermo24.py --from 08:00 --to 20:00   # дневное окно
   python3 scripts/thermo24.py --hours 24 --ymin 20 --ymax 29
   python3 scripts/thermo24.py --list                # показать доступные потоки
   python3 scripts/thermo24.py --options             # живые опции сценария (hysteresis, fanTempStep…)
   ```
   PNG сохраняется в `./charts/vt_last24h_<дата>.png` (папка создаётся; переопределить `--outdir`/`--out`).
   Запускать из корня проекта `SH_VirtualThermostat`, чтобы график лёг в `charts/`.
3. Показать график пользователю и дать короткую сводку: держался ли коридор 24.0–24.6 вокруг
   цели 24.3, провалы <23.8°, число/окна активного охлаждения, пик улицы, заряд батареи датчика.

## Прод-контекст (на 2026-06)
Сценарий v3.11.0-ac (Logic 5), цель 24.3, гистерезис 0.3, сила 3, упреждение 0.2, fanTempStep 0.2,
acModulateAtTarget вкл, обдув 8–20. Целевая кондея «инвертирована»: чем сильнее перегрев комнаты,
тем ниже setpoint шлётся в VIOMI. Открытый вопрос — поведение в жаркий день (улица 28–34°) ещё не
проверено. Батарея внешнего датчика была 24% — следить, при разряде растут паузы обновления
(timeout 60 мин → возможен ложный «потерян датчик»).
