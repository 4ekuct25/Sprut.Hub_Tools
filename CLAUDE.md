# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Назначение репозитория

Коллекция логических и глобальных сценариев для умного дома **Sprut.Hub**. Каждая папка верхнего уровня в CamelCase — отдельный сценарий (`VirtualThermostat`, `CircadianLight`, `RuntimeCounter` и т.д.). Сценарии импортируются и запускаются в самом хабе — здесь нет сборки, тестов через CI, package.json или линтера. Репозиторий — это исходники + документация.

## Среда выполнения и ограничения языка

Сценарии исполняются движком **Nashorn** внутри хаба. Это критическое ограничение, влияющее на каждую правку:

- Поддерживается **только ES5** плюс точечные возможности ES6: стрелочные функции, шаблонные строки, `const`/`let`, `Map`/`Set`, `for...of`.
- **Запрещено**: классы, модули (`import`/`export`), деструктуризация, промисы, `async`/`await`, spread/rest, `Object.assign` и прочий ES6+.
- Функции `trigger`/`compute`, несмотря на то что вызываются "асинхронно" хабом, пишутся как обычные синхронные функции — `async`/`await` нет.

Для асинхронных эффектов используются `setTimeout`/`setInterval`, `Cron`, подписки `Hub.subscribe(...)`.

## Два типа сценариев

В одной папке могут жить оба типа (например, `Battery/source/Global.js` + `Battery/source/Logic.js`, `TurnOffAllLight/source/MasterSwitch.js` (global) + `MasterSwitch.Logic.js`).

### Логический сценарий (`source/*.js`)

Реагирует на изменения характеристик конкретных устройств. Структура — см. `ScenarioTemplate/LogicScenarioTemplate.js` и `ScenarioTemplate/README.md`:

- Блок `info = { name, description, version, author, onStart, sourceServices, sourceCharacteristics, options, variables }` — **только метаданные**. Хаб читает `info` при загрузке/сохранении сценария. **Из кода в `info` обращаться нельзя**: значения опций и переменных там не будут обновляться. Все рабочие данные приходят аргументами в `trigger`/`compute`.
- `function trigger(source, value, variables, options, context)` — основная точка входа. При `onStart: true` вызывается также при запуске хаба и при каждом сохранении сценария.
- `function compute(source, value, variables, options, context)` — синхронный, возвращает значение, которое хаб запишет в характеристику. Использовать только при острой необходимости — выполняется синхронно и может затормозить хаб.
- `variables` — состояние сценария между вызовами; сбрасывается при перезагрузке хаба.

### Глобальный сценарий

Это **библиотека** функций/констант, доступная другим сценариям через `GlobalVariables.имя` или алиас `global.имя`. У него нет блока `info`, нет `trigger`/`compute` — просто набор функций верхнего уровня.

Жёсткие ограничения глобальных сценариев (не работают, если код вызван **из самого** глобального сценария напрямую; работают, если функция вызвана из логического/блочного сценария):
- нельзя изменять характеристики устройств;
- нельзя ставить таймеры/cron, которые вызывают этот же глобальный сценарий.

Практический вывод: глобальные сценарии — чистые утилиты. Управление устройствами и таймерами живёт в логических сценариях, которые вызывают глобальные функции.

## Типичная структура папки сценария

```
ИмяСценария/
  README.md                  # документация сценария (что делает, опции, переменные, ограничения)
  source/
    ИмяСценария.js           # логический и/или глобальный сценарий
  ИмяСценария.json           # экспорт сценария из UI хаба — НЕ редактировать
  .tests/
    config.json              # конфиг тестов (схема ScenarioSimulator)
    *.test.js                # тесты от спецификации README
    preset.json              # фикстуры для ручной проверки в веб-UI
```

Имена папок и `.js`-файлов — CamelCase. Допустимы исторические русские имена файлов (`Логический.js`, `Глобальный.js`) в существующих папках типа `CircadianLight/` — не переименовывать без причины.

Файлы README и комментарии в коде — на русском языке. Поля `name`/`desc` в опциях используют структуру `{ ru: "...", en: "..." }` для двуязычного UI.

### Обязательная пара: `.tests/` + `preset.json` для нового сценария

При создании любого нового сценария ВСЕГДА создаёшь:

1. **`.tests/config.json`** — со ссылкой на схему и логические/глобальные исходники.
2. **`.tests/*.test.js`** — тесты от README (см. §«Правила работы с тестами»).
3. **`.tests/preset.json`** — фикстура для ручной проверки сценария через веб-UI ScenarioSimulator. Описывает комнаты, аксессуары и сервисы, нужные для запуска. Один из аксессуаров помечается `"target": true` — тот, чьи характеристики триггерят `trigger(source, value, ...)` сценария (определяется по `info.sourceServices` / `info.sourceCharacteristics`).

Образец preset — `VirtualThermostat/.tests/preset.json`. Поля:

- `name`, `description` — для UI (русский).
- `options` — начальные значения опций сценария (имена из `info.options.*`).
- `variables` — обычно `{}`, инициализирует state.
- `rooms` — `[{"name":"..."}]`.
- `accessories[]` — `{ id, name, room, target?, services: [{ type, name?, characteristics: [{ type, value }] }] }`.
- Не указывай `time` — UI берёт реальное текущее.
- Типы сервисов/характеристик — `HS.*` / `HC.*` строки из `ScenarioTemplate/sh_types.json`.

Pre-flight перед коммитом сценария: проверь, что `.tests/preset.json` есть и `bun run cli serve` показывает сценарий в списке с рабочей ручной проверкой.

## Что нельзя редактировать или генерировать

- **`.json`-файлы сценариев в папках сценариев** (например, `VirtualThermostat/VirtualThermostat.json` и т.п.) — это экспорты из UI Sprut.Hub. Пользователь обновляет их сам. Не создавать и не править.
- Изменения, которые пользователь сделал вручную в коде, — не откатывать. Если видишь ошибку в его правке — указать на неё в тексте, а не "молча исправить".

## Источники истины по API

Когда нужны имена сервисов, характеристик, поведение API хаба:

- **`ScenarioTemplate/README.md`** — каноническое описание структуры сценариев и принципов кода ("Минимум кода, максимум читаемости. Читаемость в приоритете", SRP, DRY, KISS, ранние возвраты, без магических чисел).
- **`ScenarioTemplate/LogicScenarioTemplate.js`** — готовый шаблон логического сценария со всеми типами опций.
- **`ScenarioTemplate/spruthub.js`** — JS API хаба с TypeScript-объявлениями и русской документацией: `Hub`, `Accessory`, `Service`, `Characteristic`, `Cron`, `Notify`, `SSH`, `Mail`, `HttpClient`, `Utils`, `UtilsNet`, `setTimeout`/`setInterval`/`clear`, перечисления `HC` и `HS`.
- **`ScenarioTemplate/sh_types.json`** — полный список сервисов и характеристик с допустимыми форматами и значениями. Для промптов использовать сжатые версии: `prompt_optimized_types.txt` (95% сжатия) или `compressed_types.txt`. Скрипты сжатия (`compress_types.js`, `ultra_compact_types.js`, `prompt_optimized_types.js`) запускаются через `node <имя>.js`, см. `ScenarioTemplate/COMPRESSION_README.md`.

Структура устройства: **Аксессуар → Сервисы → Характеристики**. У сервиса каждый тип характеристики может быть только один; список разрешённых характеристик для типа сервиса задан в `sh_types.json`.

`source.getUUID()`: для аксессуара — `"123"`, для сервиса — `"123.45"`, для характеристики — `"123.45.67"`.

## Тестирование

Два подхода:

### Внешний тест-раннер `ScenarioSimulator/` (предпочтительный)

Эмулятор хаба + CLI + веб-UI. Тесты живут в `.tests/` рядом со сценарием, прогоняются на macOS/Linux/Windows без хаба.

```bash
cd ScenarioSimulator && bun install && bun run generate
bun run cli run TurnOffAllLight --root ..      # запуск тестов сценария
bun run cli serve --port 5173 --root ..         # веб-интерфейс
```

Полная документация — `ScenarioSimulator/README.md`. Формат конфига и DSL описаны там же.

#### Правила работы с тестами

1. **После любой правки сценария — прогнать тесты сценария**: `bun run cli run <Name> --root ..`. Перед коммитом — полный прогон `bun run cli run --root ..`. Все правки сценария должны оставлять тесты зелёными, иначе нужно либо чинить код, либо обновить тест (если правка меняет публичный контракт — тогда сначала обновляется README сценария, потом тест, потом код).
2. **Тесты пишутся ОТ СПЕЦИФИКАЦИИ (README/доки), а не от текущей реализации**. Каждый `describe` соответствует разделу README, каждый `it` — конкретному утверждению. Если тест проходит только потому что он повторяет логику реализации — он бесполезен. Тест должен падать, если код перестаёт делать то, что обещано в README.
3. **Если тест упал — сначала разобраться, кто неправ**: код, README или сам тест. Не подгоняй тест под код "чтобы стало зелёным" — это маскирует реальные расхождения README↔код.
4. **Покрытие — отдельно global и logic части**. Если у сценария есть и `*.Global.js`, и `*.Logic.js`, для каждого свой `.test.js` файл (см. `TurnOffAllLight/.tests/master-switch-global.test.js` + `master-switch-logic.test.js`). Логический тестируется через `scenario.run({source, value, variables, options, context})`; глобальный — через `scenario.call(name, args)` либо через подмену `variables.global.X = spy`.
5. **Тесты — артефакт первого класса, поддерживай в актуальном состоянии**. При добавлении опции в сценарий — добавь тесты на её работу. При удалении — удали соответствующие тесты. README, код, тесты должны быть синхронны.

### Старый подход через `UnitTest/source/UnitTests.js` (поддерживается)

Прогон внутри хаба, тесты встроены в сценарий:

- Скопировать функцию `runTests` (и нужные `assert*`) в свой сценарий и вызвать в `trigger`. При `onStart: true` `trigger` вызывается на каждом сохранении сценария — это и есть прогон тестов.
- Для тестов без реальных устройств — мок-объекты через `createUnitTestFullAccessory`/`createUnitTestAccessory`/`createUnitTestService`/`createUnitTestCharacteristic`. Возвращают объекты, совместимые по API с настоящими `Accessory`/`Service`/`Characteristic`.

Новые сценарии лучше тестировать через `ScenarioSimulator/`, существующие тесты можно мигрировать командой `scenario-sim migrate <scenario>`.

## README сценария — обязательные разделы

Когда пишешь или правишь `README.md` сценария — должно быть:
- назначение и поведение;
- ключевые опции (`options`) и переменные (`variables`) с объяснением каждого поля;
- особенности настройки в UI Sprut.Hub;
- известные ограничения и важные замечания.
