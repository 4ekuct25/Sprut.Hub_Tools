# Инструкции для Claude — форк VirtualThermostat

Форк сценария Sprut.Hub: виртуальный термостат управляет кондиционером (VIOMI Cross 18000BTU)
по внешнему датчику (eWeLink SNZB-02D). Версии форка помечаются суффиксом `-ac`.

## ⚠️ ЧЕК-ЛИСТ ЛЮБОГО ИЗМЕНЕНИЯ КОДА (выполнять ВСЁ, в одном коммите)

При КАЖДОМ изменении `source/VirtualThermostat.js` обязательно и за один проход:

1. **Поднять версию** `version: "X.Y.Z-ac"` в `source/VirtualThermostat.js` (поле `info.version`).
2. **Обновить README.md** — ВСЕГДА, это не опциально:
   - версия в заголовке `## 🔀 Форк: управление кондиционером (vX.Y.Z-ac)`;
   - при новой опции — строка в таблице «Новые параметры» и, если нужно, в разделе «Как работает» / «Синхронизация по питанию»;
   - запись в «🕘 История изменений»: версии идут от СТАРОЙ к НОВОЙ, **новая версия добавляется В КОНЕЦ списка** (последней); метка «(текущая, форк)» — только у самой новой (последней) версии, со старой её снять. Не вставлять новую версию в середину и не дублировать существующие.
3. **Тесты** — добавить кейсы на новое поведение в `.tests/*.test.js`, прогнать ВЕСЬ набор, добиться 0 failed.
4. **Пересобрать шаблон** `VirtualThermostat.json` из исходника (поле `scenarioTemplate.data` = содержимое `source/VirtualThermostat.js`).
5. **Коммит** одним commit'ом (код + README + тесты + json). `git push` делает пользователь сам.

Терминология в опциях и документации: «целевая температура», НЕ «уставка».

## Прогон тестов

ScenarioSimulator (bun) из апстрим-клона:
```
export PATH=$HOME/.bun/bin:$PATH
cd /tmp/Sprut.Hub_Tools/ScenarioSimulator   # клонировать апстрим, если нет
bun run packages/cli/bin/scenario-sim.ts run "<путь к папке VirtualThermostat>"
```
Аргумент — позиционный путь к папке сценария (НЕ `--config`). В тестах фейковое время — фикстура
`time` (`time.now()`, `time.tick(ms)`, `time.set(iso)`), НЕ реальный `Date.now()`.

## Грабли среды Sprut.Hub (проверено на живом железе)

- **JS-движок — Nashorn.** В шаблонных строках `` `...` `` нельзя `/` рядом с `${}` — парсится как regex
  (ScriptException). Деление/слэши возле интерполяции выносить в переменную.
- **setValue асинхронный** — немедленный `getValue()` после записи возвращает СТАРОЕ значение.
  Никаких read-back проверок «применилось ли»: дадут ложный отказ (в симуляторе синхронно, тесты не ловят).
- **Значения характеристик — Java-объекты.** Сравнивать через `==` и `toNum()`/`toBool()`, не `===`.
- **`variables` в колбэках подписок** захватываются на момент создания подписки и могут быть несвежими;
  решающие проверки делать stateless по живым характеристикам (`computeDesiredAcState`). После смены
  опций пользователю нужно дёрнуть «Активно» выкл/вкл, чтобы подписки пересоздались.
- **Переигрывание характеристик** при пересохранении настроек / старте хаба идёт с контекстом
  `LOGIC[...] <- WEB[...]` БЕЗ звена `C[...]` (или `HUB[OnStart]`). Это НЕ действия пользователя
  (настоящие: `LOGIC <- C[...] <- WEB`). См. `isSystemReplayContext`.
- **VIOMI игнорирует режим «Выключено»** в термостат-сервисе (22.17) — возвращает 2. Выключение только
  через сервис питания «Кондиционер» (22.13, FanBasic.On) — опция `acPowerSwitch`.
- **Запоздалые подтверждения VIOMI** — устройство переизлучает старое состояние через несколько секунд
  после команды. Окно подавления `AC_ECHO_WINDOW_MS` (30 с) + мягкие переотправки (`AC_REASSERT_MAX`).
- **Zigbee-датчики шлют данные только при изменении** — при стабильной температуре молчат часами. Молчание
  онлайн-датчика (характеристика `C_Online`) отказом НЕ считать; отказ = уход в офлайн. См. `isSensorOnline`.

## Карта устройств в хабе пользователя

- Виртуальный термостат: MakeSimple Virtual, сервис Thermostat — `42.13`. Сценарий в хабе — №5.
- Датчик: eWeLink SNZB-02D, TemperatureSensor — `8.13`.
- Кондиционер: VIOMI Cross 18000BTU. Режим (Thermostat) — `22.17`, питание (FanBasic.On) — `22.13`.
- Доступ: web.spruthub.ru (логин у пользователя). Финальные `git push` — пользователь сам.

## Режимы кондиционера VIOMI (карта)

`TargetHeatingCoolingState` кондея (22.17.20) расширенный: **−2 = Осушитель, −1 = Вентилятор**, 0 = Выкл, 1 = Нагрев, 2 = Охлаждение, 3 = Авто. Сценарий сам ставит только 0/1/2/3. Появление −1/−2 = выбор пользователя → сценарий обязан отступить (acManualOverride), а не пытаться регулировать (иначе пинг-понг питания, см. v3.8.0-ac).
